# 上传清洗与高价值复盘工作流契约

本文档冻结当前产品与技术边界。后续维护不得重新引入旧归因页、渠道下钻页、维护台账页、历史批次重算入口或一次性交接脚本作为产品主流程。

## 1. 产品入口

Streamlit 只保留五个页面：

- `总览`
- `上传清洗`
- `高价值内容复盘`
- `本地总表/素材缓存`
- `历史趋势`

季度和年度能力保留在 `上传清洗` 页：既允许直接上传季度/年度数据，也允许根据本地月数据生成季度/年度汇总。

## 2. 主流程

1. 上传一个周期的 Excel、CSV、ZIP 或文件夹。
2. 系统按周期和渠道保存原始源文件。
3. 同周期同渠道再次上传时，用户确认后覆盖该渠道旧源文件。
4. 系统标准化字段、识别单行渠道总数据、同渠道内去重聚合。
5. 系统读取线上飞书台账并更新本地 `content_assets`，再对本周期缺失字段做匹配回填。
6. 清洗结果写入 SQLite，并输出核验 workbook。
7. 用户在 `高价值内容复盘` 页触发素材缓存、缺失补采和多模态复盘。
8. 复盘完成后，新素材 manifest、多模态结果和类型聚合写入本地库，后续周期优先复用。

清洗和复盘必须能分开重跑：清洗成功不依赖多模态复盘，复盘失败不破坏已清洗数据。

## 3. 运行目录

- `data/weeks/`：周度原始源文件目录。
- `data/months/`：月度原始源文件目录。
- `data/quarters/`：直接上传的季度原始源文件目录。
- `data/years/`：直接上传的年度原始源文件目录。
- `processed/`：清洗产物、核验 workbook 和必要诊断文件。
- `.runtime/workflow.sqlite3`：权威运行库。
- `.runtime/top-assets/{platform}/{stable_asset_key}/`：本项目复用或补采的素材缓存，同一素材跨周期只存一份。
- `outputs/`：仅保留给显式命令行导出模式；Streamlit 主流程不依赖它。

## 4. 渠道和字段配置

默认渠道必须覆盖：

- 小红书商业化
- 小红书市场部
- 抖音市场部
- 抖音商业化
- B站市场部
- B站商业化
- 微信市场部
- 微信商业化

新增平台和渠道必须通过 `config/channel_profiles.yml` 接入，不要求改代码。字段别名优先于通用字段映射；未映射的原始字段按原表头保留，避免排查时丢失来源信息。

字段匹配顺序：

1. 渠道配置中的字段别名。
2. `config/field_mapping.yml` 的通用字段别名。
3. 原始列名。

## 5. 清洗结果字段

本周期素材明细固定保留：

- 周期
- 渠道
- 平台 ID
- 账号
- 标题
- tag 词
- 抖音/小红书一级内容类型
- 抖音/小红书二级内容类型
- B站内容类型
- 链接
- 消耗
- 曝光
- 激活数
- 付费数
- 激活成本
- 付费成本

平台 ID 规则：

- 抖音使用标准 URL 作品 ID；分享链接必须尽量解析为标准 URL、作品 ID、标题、封面和视频等字段。
- B站使用 BV 号。
- 小红书使用笔记 ID。

如果渠道 Excel 只有一行数据，视为渠道总数据，只进入 `period_channel_totals` 和总览，不进入素材明细、去重、匹配或高价值复盘。

## 6. SQLite 表

- `content_performance_items`：本周期清洗后素材明细。
- `period_channel_totals`：单行渠道总数据和聚合总览数据。
- `content_assets`：本地总素材表。
- `harvester_asset_manifests`：素材缓存和采集结果。
- `multimodal_recap_items`：每个高价值素材的多模态复盘结果。
- `type_recap_items`：按内容类型聚合后的复盘结果。
- `recap_settings`：当前默认价值权重，至少包含 `activation_weight`、`first_pay_weight`、`updated_at`。

## 7. 去重和匹配

同渠道内去重优先级：

1. 稳定内容 ID，例如 B站 BV、小红书笔记 ID、抖音作品 ID。
2. 规范化内容链接。
3. 规范化标题。

数值指标累加，成本和率用合计值重新计算。不跨渠道合并。

飞书同步与匹配：

- 线上飞书只读，不自动写回。
- 从 harvester 飞书配置读取线上台账，同步到本地 `content_assets`。
- 抖音按标题模糊匹配。
- 小红书按笔记 ID 精确匹配。
- B站按 BV 号精确匹配。
- 匹配成功后只回填本周期缺失字段，不覆盖上传源已有有效值。

## 8. 高价值内容复盘

只分析抖音、小红书、B站：

- 抖音：按渠道消耗 Top20、曝光 Top20。
- 小红书/B站：按渠道消耗 Top10、曝光 Top10。
- 任一平台单素材消耗 > 2000 或曝光 > 100000，也进入高价值池。

素材来源优先级：

1. harvester 每日采集保留的素材缓存和 manifest。
2. 本项目已复制的 `.runtime/top-assets/{platform}/{stable_asset_key}/`。
3. 本项目 TopN 补采任务，只补缺。

类型复盘必须包含：

- 抖音一级/二级内容类型占比。
- 小红书一级/二级内容类型占比。
- B站内容类型占比。
- 每类曝光、消耗、激活、付费、激活成本、付费成本、价值。

价值公式：

```text
价值 = 激活数 * 激活权重 + 付费数 * 付费权重
```

激活权重和付费权重由用户在页面填写；填写后保存为默认值，直到下一次更新。

抖音只有一级内容类型且为“长视频”或“说唱”时，二级内容类型沿用一级内容类型。

第一版环比只基于本地上一周期数据；联网时间原因分析暂不实现。

## 9. 环境配置

`env_bridge` 只从 harvester `.env` 补齐本项目缺失的 Feishu/MiniMax 配置，不覆盖本项目已有值。

MiniMax 第一版只用于多模态内容理解和总结；不能把联网原因分析作为验收要求。

## 10. 验收

上线前至少验证：

1. 字段别名映射正确。
2. 同渠道重复内容合并，指标相加，成本重算。
3. 单行渠道总数据只进总览。
4. 抖音分享链接解析失败时进入待补齐。
5. 飞书同步本地总表后，抖音/小红书/B站匹配回填不覆盖有效字段。
6. 高价值池 TopN/阈值筛选正确。
7. harvester 每日缓存优先复用，本地已有缓存不补采。
8. 类型聚合指标和价值权重正确。
9. 导航只保留五个页面。
