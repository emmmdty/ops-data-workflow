# Excel 清洗上线工作流契约

本文档冻结统一上线改造的产品与技术契约，供后续并行实现线程使用。后续实现必须优先满足本文档，不得重新引入旧 `data/raw`、`archive`、`outputs` 报告下载口径。

## 1. 目标闭环

系统面向不懂技术的运营同事，主流程固定为：

1. 上传一个周期的 Excel、CSV、ZIP 或文件夹。
2. 系统按周期和渠道保存原始源文件。
3. 同周期同渠道再次上传时，覆盖该渠道旧源文件。
4. 系统重新清洗并生成一个统一清洗 workbook。
5. 人工只审核高消耗、高风险冲突、补齐失败的少量内容。
6. 审核保存后，清洗 workbook、SQLite 页面数据、图表和 AI 复盘输入同步更新。
7. 用户手动触发 AI 复盘报告生成。

页面模式下，统一清洗 workbook 是唯一业务交付物；`outputs/` 不作为默认产物目录。

## 2. 运行目录契约

- `data/months/`：月度原始源文件目录。
- `data/weeks/`：周度原始源文件目录。
- `data/reference/`：人工维护投稿台账、映射表和参考表。
- `.runtime/workflow.sqlite3`：运行库，保存批次、canonical 明细、审核、图表和 AI 复盘结果。
- `processed/{period_key}/{batch_id}/`：运行产物目录，保存统一清洗 workbook、清洗清单和必要诊断文件。
- `outputs/`：仅保留给显式命令行文件导出模式；Streamlit 页面生成流程不得依赖它。

废弃项：

- 旧 `data/raw` 运行入口。
- `archive_root` 和旧 `archive/` 输入归档口径。
- `output/playwright` 页面测试残留目录。
- `legacy-output-only` 作为主要业务入口。
- 未被页面使用的 raw/file backup 操作。
- README 或 UI 中对外展示的 `三级题材`、`内容明细`、`数据质量` 旧页面口径。

## 3. 上传输入契约

支持输入：

- 单个或多个 `.xlsx`、`.xls`、`.csv` 文件。
- ZIP 压缩包。
- Streamlit 文件夹上传，保留子目录用于周期识别。

周期识别：

- 优先从上传路径或目录名识别周度、月度、季度、年度。
- 周度源文件写入 `data/weeks/YYYYMMDD-YYYYMMDD/`。
- 月度源文件写入 `data/months/YYYYMM/`。
- 识别失败时，使用页面上选择的周期。

同周期同渠道覆盖：

- 渠道由渠道配置契约识别。
- 若目标周期已有同渠道源文件，默认阻止写入并提示用户确认。
- 用户确认覆盖后，删除该渠道旧源文件，写入新源文件，失效该周期旧清洗 workbook 和清洗 manifest。
- 覆盖只影响该周期该渠道，不影响其他渠道、其他周期和生产运行库外的文件。

## 4. 渠道配置契约

默认渠道必须覆盖：

- 小红书商业化
- 小红书市场部
- 抖音市场部
- 抖音商业化
- B站市场部
- B站商业化
- 微信市场部
- 微信商业化

新增平台和渠道必须通过配置接入，不要求改代码。配置来源为 `config/channel_profiles.yml`，并在维护台账页面提供中文编辑入口。每个渠道配置项至少包含：

- `channel`：标准渠道名。
- `platform`：平台名，例如 小红书、抖音、B站、微信。
- `group`：内部兼容字段，默认与平台名一致；不作为同事可见字段输出。
- `filename_keywords`：文件名或 sheet 名关键词。
- `field_aliases`：该渠道额外字段别名。
- `account_filter_enabled`：是否启用账号过滤。
- `active`：是否启用该渠道。

字段匹配顺序：

1. 渠道配置中的字段别名。
2. `config/field_mapping.yml` 的通用字段别名。
3. 原始列名保留为渠道独有字段。

## 5. 统一清洗 Workbook 契约

每次成功生成页面数据时，必须在 `processed/{period_key}/{batch_id}/` 下生成一个统一清洗 workbook，建议文件名为：

```text
cleaned_channels.xlsx
```

workbook 结构：

- 一个渠道一个业务 sheet，sheet 名等于标准渠道名。
- 系统 sheet 固定为：`导入日志`、`重复内容`、`冲突项`、`补齐来源`、`审核记录`。
- 如 sheet 名超过 Excel 限制或重复，使用稳定短名，并在 `导入日志` 记录原渠道名。
- 文件名无法识别到已有渠道时，不阻塞清洗；平台写为 `其他`，渠道写为文件名 stem，并正常生成独立业务 sheet。

每个渠道 sheet 的前置统一字段固定为：

- 周期
- 平台
- 渠道
- 账号
- 内容形式
- 内容类型
- 标题
- 内容ID
- 素材ID
- 唯一标识
- 内容链接
- 发布时间
- 消耗
- 曝光量
- 点击量
- 激活数
- 付费数
- 激活成本
- 付费成本
- 补齐来源
- 补齐置信度
- 复核状态
- 复核原因

统一字段之后追加渠道独有字段。未映射的原始字段按原表头保留，例如原始表头为 `7日付费成本`，渠道 sheet 中也输出 `7日付费成本`。

每个渠道 sheet 只追加该渠道实际出现且至少有非空值的未映射字段；已映射到统一字段的原始列不重复输出。禁止丢弃未知原始字段，除非它是明确的系统临时文件、空列、重复副本或已在配置中标记忽略。

## 6. 清洗、去重、补齐契约

去重优先级：

1. 同渠道内稳定内容 ID，例如 B站 BV、 小红书笔记 ID、抖音作品 ID。
2. 同渠道内规范化内容链接。
3. 同渠道内规范化标题。

去重规则：

- 数值指标累加：消耗、曝光、点击、激活、付费。
- 成本和率重新计算，不取行均值。
- 文本字段优先保留非空、来源置信度更高、标题信息更完整的值。
- 同一 ID 但标题、链接、内容类型冲突时，进入 `冲突项` 和审核队列。
- 不跨渠道合并，除非后续明确增加跨渠道归因功能。

补齐优先级：

1. 原始 Excel 字段。
2. 渠道配置字段别名。
3. 通用字段别名。
4. 投稿台账和历史映射。
5. 平台公开信息补齐。
6. AI 候选。
7. 人工审核。

平台公开补齐要求：

- B站：可使用公开 API、已有缓存和 BV 号解析。
- 抖音：可使用短链、作品 ID、标题和缓存解析；不得要求登录态。
- 小红书：可使用 note ID、链接和可公开解析字段；不得要求登录态。
- 遇到风控、登录、验证码或请求失败时，字段留空，记录失败原因，不阻塞清洗。

补齐必须写入 `补齐来源` sheet，字段至少包含：

- batch_id
- channel
- content_id
- material_id
- title
- field_name
- old_value
- new_value
- source
- confidence
- status
- reason

## 7. 审核队列契约

审核队列只包含少量必须人工处理的数据，每个批次最多 49 条。默认优先级为：

- 高消耗且存在高风险冲突：同 ID 标题冲突、同标题多链接、投稿台账多候选、内容类型冲突。
- 高消耗且关键字段补齐失败：内容链接、内容 ID、标题、内容类型。
- 每周期每渠道消耗 Top 20 中仍有关键字段补齐失败。
- 其他单条消耗大于等于 2000 元的内容。

普通低消耗、低风险的 AI 分类结果不得全部进入审核队列。完整冲突、补齐来源和失败原因保留在清洗 workbook 的系统 sheet 与 canonical 诊断字段中。

审核保存后必须：

- 写入审核记录。
- 更新分类映射或字段修正。
- 重新生成该批次最终 canonical。
- 重新生成统一清洗 workbook。
- 刷新图表和 AI 复盘输入。

## 8. 图表与 AI 复盘输入契约

图表和 AI 复盘只读取审核后的最终 canonical/SQLite 结果。

禁止：

- 从旧 `outputs/{batch_id}/analysis.xlsx` 读取页面数据。
- 从历史 `report.html` 或 `canonical.csv` 反推图表。
- 使用未审核的临时分类结果生成最终 AI 复盘。

允许：

- 总览、渠道页、历史趋势共用 SQLite 批次数据。
- 手动 AI 复盘读取最终汇总、渠道表现、重点内容、高消耗审核结果和图表摘要。

## 9. 后续线程边界

线程 A：产物与数据模型。

- 修改 `ops_data_workflow/channel_clean.py`、`ops_data_workflow/models.py`、`ops_data_workflow/workflow.py`。
- 交付统一清洗 workbook 生成能力。
- 不修改 Streamlit UI。

线程 B：渠道和字段配置。

- 新增 `config/channel_profiles.yml` 和渠道配置加载模块。
- 修改渠道识别和维护台账页面。
- 不修改清洗去重策略。

线程 C：清洗、补齐、去重。

- 修改 `ops_data_workflow/raw_cleaning.py`、`ops_data_workflow/pipeline.py`、`ops_data_workflow/content_ledger.py`、`ops_data_workflow/content_metadata.py`。
- 交付补齐来源记录和收敛后的冲突标记。
- 不修改页面布局。

线程 D：审核收敛与图表衔接。

- 修改审核队列构建、`ops_data_workflow/review_resolutions.py`、`app.py` 中内容审核页。
- 确保审核保存后刷新最终 canonical、清洗 workbook、图表和 AI 复盘输入。
- 不改渠道配置格式。

集成线程：

- 合并 A-D。
- 统一 README、`docs/同事使用说明.md`、局域网启动和故障排查说明。
- 清理废弃入口和旧文案。

## 10. 上线验收

上线前必须在独立 worktree 中验证：

1. `python -m pytest -q`
2. 使用临时目录或复制样例数据上传七个默认渠道。
3. 同周期同渠道覆盖后只替换该渠道。
4. 生成一个统一清洗 workbook，且每个渠道一个 sheet。
5. 审核队列明显少于全量行数，只包含高消耗和高风险项。
6. 审核保存后图表和 AI 复盘输入更新。
7. 使用非 8501 端口预演局域网启动；确认后再安排 8501 上线窗口。

不得在同事正在使用的生产 checkout 中直接验证破坏性流程。
